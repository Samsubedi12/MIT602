Phase 1: AWS Infrastructure Setup

This phase covers the foundational steps of preparing the cloud environment on AWS. Here, we launch the virtual server, configure network access rules, and assign a static IP address for stable connectivity.

  1. Navigate to the EC2 Dashboard in the AWS Console.
  2. Click "Launch instances".
  3. Select the "Ubuntu Server 22.04 LTS" AMI.
  4. Choose the `t2.micro` instance type (Free Tier eligible).
  5. Configure a new Security Group with rules for SSH (Port 22 from My IP), HTTP (Port 80 from Anywhere), and HTTPS (Port 443 from Anywhere).
  6. Create and download a new key pair (e.g., `moodle-key-pair.pem`). This is essential for SSH access.
  7. Launch the instance.

The inbound rules were configured during instance launch as part of the Security Group setup. This is a critical step to control who can access the server and on which ports. The rules are:

  • SSH (Port 22): Source set to "My IP" to ensure only you can manage the server.
  • HTTP (Port 80): Source set to "Anywhere" to allow users to access the site.
  • HTTPS (Port 443): Source set to "Anywhere" for secure user access.

An Elastic IP is a static public IPv4 address. It's important because the default public IP of an EC2 instance can change if it's stopped and started.

  1. In the EC2 Dashboard, go to "Elastic IPs".
  2. Click "Allocate Elastic IP address".
  3. Select the new IP, click "Actions", and "Associate Elastic IP address".
  4. Choose your `MoodleServer` instance and click "Associate".

Phase 2: Moodle Installation & Configuration

With the server running, this phase details the installation of the software stack required for Moodle: the Apache web server, the MariaDB database, and PHP. It concludes with the Moodle application setup itself.

First, update your server's packages:

sudo apt update && sudo apt upgrade -y

Install Apache, MariaDB, and PHP with required extensions:

sudo apt install apache2 mariadb-server mariadb-client -y
sudo apt install php libapache2-mod-php php-cli php-fpm php-json php-curl php-gd php-intl php-mbstring php-mysql php-xml php-zip php-soap php-opcache -y

Secure your MariaDB installation:

sudo mysql_secure_installation

Log in to MariaDB and create a dedicated database and user for Moodle.

sudo mysql -u root -p

CREATE DATABASE moodle DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
CREATE USER 'moodleuser'@'localhost' IDENTIFIED BY 'YourStrongPasswordHere';
GRANT ALL PRIVILEGES ON moodle.* TO 'moodleuser'@'localhost';
FLUSH PRIVILEGES;
EXIT;

Download Moodle, unzip it, and create a secure data directory.

cd /var/www/html/
sudo wget https://download.moodle.org/download.php/direct/stable404/moodle-4.4.zip
sudo unzip moodle-4.4.zip -d /var/www/html/
sudo mkdir /var/www/moodledata

Set the correct ownership and permissions for the web server to manage Moodle's files.

sudo chown -R www-data:www-data /var/www/html/moodle
sudo chown -R www-data:www-data /var/www/moodledata
sudo chmod -R 755 /var/www/html/moodle
sudo chmod -R 777 /var/www/moodledata

After setting permissions, navigate to `http://<Your_Elastic_IP>/moodle` in your browser and follow the on-screen instructions to complete the installation.

Phase 3: Securing the Instance

Security is not an afterthought. This phase focuses on hardening the server by encrypting web traffic with SSL, configuring a host-based firewall, creating a non-root admin user, and locking down SSH access.

Install Certbot and request a free SSL certificate to enable HTTPS. This requires a domain name pointing to your server's Elastic IP.

sudo apt install certbot python3-certbot-apache -y
sudo certbot --apache -d your-domain.com

Enable the Uncomplicated Firewall (UFW) on the server to provide another layer of network security.

sudo ufw allow ssh
sudo ufw allow http
sudo ufw allow https
sudo ufw enable

Avoid using the root user for daily tasks. Create a new user and give it `sudo` privileges.

sudo adduser myadminuser
sudo usermod -aG sudo myadminuser

Disable password-based login for SSH to prevent brute-force attacks. This makes your server significantly more secure.

  1. Add your SSH public key to `~/.ssh/authorized_keys` for the new admin user.
  2. Edit the SSH configuration file: `sudo nano /etc/ssh/sshd_config`
  3. Set `PasswordAuthentication no` and `PermitRootLogin no`.
  4. Restart the SSH service: `sudo systemctl restart sshd`

Important: Test your key-based login in a new terminal window *before* closing your current session!

Phase 4: Backup and Monitoring

A reliable service requires robust operational procedures. This final phase covers setting up automated backups for your Moodle data and files, and outlines key logs and services for monitoring system health and security.

Create simple bash scripts to back up the Moodle database and files, then automate them with a cron job to run daily.

Database Backup Script (`/usr/local/bin/backup_db.sh`):

#!/bin/bash
DB_NAME="moodle"
DB_USER="moodleuser"
DB_PASS="YourStrongPasswordHere"
BACKUP_DIR="/var/backups/moodle_db"
TIMESTAMP=$(date +%Y%m%d%H%M%S)
mkdir -p ${BACKUP_DIR}
mysqldump -u ${DB_USER} -p${DB_PASS} ${DB_NAME} > "${BACKUP_DIR}/${DB_NAME}_${TIMESTAMP}.sql"
find ${BACKUP_DIR} -type f -mtime +7 -delete

Schedule with Cron (`sudo crontab -e`):

0 2 * * * /usr/local/bin/backup_db.sh
30 2 * * * /usr/local/bin/backup_moodle_files.sh

Leverage AWS CloudWatch to monitor your instance's health and performance.

  • CloudWatch Metrics: View default metrics like CPU Utilization, Network I/O, and Disk I/O in the CloudWatch console.
  • CloudWatch Alarms: Set up alarms to get notified via email or SMS if key metrics cross a defined threshold (e.g., CPU > 80% for 5 minutes).
  • CloudWatch Logs: Install the CloudWatch Agent on the EC2 instance to centralize application and system logs for easier analysis and troubleshooting.

Regularly review logs to monitor for errors and suspicious activity. Key log files include:

  • Apache Logs: `/var/log/apache2/access.log` & `error.log` for web traffic and server errors.
  • System Auth Logs: `/var/log/auth.log` for SSH login attempts and `sudo` usage.
  • MariaDB Logs: `/var/log/mysql/error.log` for database errors.
  • Moodle Logs: Accessible within the Moodle admin interface for application-level activity and errors.
